# Claude Code 全局配置

## Prerequisites 前置条件

### Node.js 版本要求
- **必须**：执行 `node --version` 确认版本 >= 18.0.0
- 如果版本过低，运行 `nvm install 18 && nvm use 18` 升级
- npm 包安装、Node.js 工具运行时**必须先检查版本**

### 网络与连接
- Git 操作前确认网络连接正常
- 大仓库操作前告知用户可能的超时风险

## 核心原则

### 1. 上下文效率优先
- **精准搜索**：使用 Grep/Glob 进行定向搜索，避免盲目读取文件
- **计划先行**：读取超过 3 个文件前，先说明计划并等待确认
- **工具优先**：优先使用专门的 Read/Grep/Glob 工具，而非 bash 命令
- **Token 意识**：大文件使用 head/tail 预览，只在必要时完整读取

### 2. 渐进式交付
- **方案先行**：复杂任务先提供简要方案（3-5 句话），等待确认后再实现
- **最小实现**：只实现明确要求的功能，避免过度工程
- **分块输出**：大文件使用 Write + Edit 组合，避免一次性输出过多内容
- **迭代优化**：先实现核心功能并验证，再考虑优化

### 3. 验证驱动开发
- **立即验证**：代码修改后立即运行测试或执行验证命令
- **语法检查**：使用 getDiagnostics 检查语法和类型错误
- **结果确认**：重要操作后用命令验证（如 ls -l, wc -l, head）
- **失败快速**：如果方法失败两次，立即切换策略，不要循环重试

## 代码质量标准

### Python 规范
- 遵循 PEP 8：4 空格缩进，snake_case 命名
- 类型注解：函数签名必须包含类型提示
- 文档字符串：公共函数/类必须有 docstring
- 错误处理：关键路径必须有 try-except
- 性能优化：优先使用向量化操作（NumPy/JAX）而非循环

### 代码组织
- 函数职责单一，每个函数不超过 50 行
- 使用描述性变量名（robot_pose 而非 rp）
- 避免不必要的类，优先使用函数式编程
- 模块化设计，便于测试和维护

### 测试要求
- 修改 auth 相关文件、MCP server 配置、集成代码后**必须运行测试**
- MCP server 集成：先用简单命令测试连接，再进行完整操作
- 验证通过后再报告完成

## 文件和版本管理

### Git 优先策略
- **使用 git 进行版本控制**，不创建重复文件（如 file_v1.py, file_v2.py）
- 重要里程碑：创建 git tag 或分支
- 实验性功能：使用 git worktree 创建独立工作区
- 提交信息：清晰描述改动的目的和影响

### 文件命名
- 使用英文和下划线：data_processor.py
- 实验性代码可加后缀：model_experimental.py
- 避免版本号后缀，让 git 管理版本

### Git 工作流
- **push 验证**：git push 必须确认成功才算完成
- push 超时：立即重试一次后再报告失败
- 大仓库操作前提醒用户超时风险
- 使用 `git status` 检查未提交更改

## 文档管理策略

### 何时创建文档
**必须创建：**
- 用户明确要求
- 复杂问题的解决方案（供未来参考）
- 架构设计和技术方案
- 论文分析和技术思路整理

**不需要创建：**
- 简单的代码修改
- 一次性问题的解决
- 常规功能实现

### 文档组织
```
docs/
├── architecture/      # 架构设计
├── papers/           # 论文分析
├── technical_ideas/  # 技术思路
├── tutorials/        # 教程文档
└── troubleshooting/  # 问题解决记录
```

### 文档命名
- 使用描述性英文名称
- AI 生成的文档可加 _AI 后缀（可选）
- 示例：transformer_analysis_AI.md, system_architecture.md

## 权限配置

### 工具限制
- 使用 `--allowedTools` 白名单特定工具用于 headless/CI 工作流
- **注意**：系统无法在命令级别限制操作（如禁止 Bash 中的 delete），仅支持工具级别限制

## Skills 系统使用

### 优先使用的 Skills
- **brainstorming**：任何创造性工作前必须使用
- **systematic-debugging**：遇到 bug 或测试失败时使用
- **test-driven-development**：实现新功能时使用
- **verification-before-completion**：声称完成前必须验证
- **writing-plans**：将设计转化为可执行计划

### Skills 触发规则
如果有 1% 的可能性某个 skill 适用，就应该调用它。Skills 提供结构化的工作流程，比临时方案更可靠。

### Skills 目录
本项目包含三类 Skills：
- `claude_code_skills/` - Claude Code 专用 skills
- `opencode_skills/` - OpenCode 专用 skills
- `opencode_superpowers/` - 通用 superpowers skills

## 项目特定配置

### 机器人和强化学习项目
- 确保代码与仿真环境兼容
- 使用不可变数据结构（适配 JAX/并行计算）
- 传感器数据处理必须包含错误处理
- 奖励函数设计要有清晰的数学依据

## 语言偏好
- 使用简体中文进行交流和文档编写
- 代码注释可以使用中文（提高可读性）
- 变量名和函数名使用英文

## 响应策略

### 分析类请求
先提供简要方案（in-chat），等待确认后再实现

### 实现类请求
1. 理解需求（必要时提问澄清）
2. 说明实现计划（3-5 句话）
3. 实现并验证
4. 报告结果

### 文件写入可靠性
- 写入目标文件一次
- 验证文件存在（ls -l 或 wc -l）
- 如果写入失败两次，切换到 Edit 或 cat heredoc
- 简要报告状态，不循环重试

## 效率提升技巧

### Token 优化
- 使用 `head -n 20 file.py` 预览文件
- 使用 `grep -n "pattern" *.py` 精准定位
- 使用 `wc -l` 获取文件行数
- 避免读取大型日志或数据文件

### 并行工作
- 独立任务使用 dispatching-parallel-agents skill
- 实验性开发使用 git worktrees
- 多个文件的独立修改可以并行执行

### 迭代开发
- 小步快跑：先实现核心功能
- 每个功能点验证后再继续
- 避免一次性实现所有功能
